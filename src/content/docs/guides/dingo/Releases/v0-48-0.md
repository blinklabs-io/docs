## v0.48.0 (May 26, 2026)

**Title:** More resilient Mithril downloads, tunable backfill behavior, and safer live sync near the chain tip

**Date:** May 26, 2026

**Version:** v0.48.0

Hi folks! Here’s what we shipped in v0.48.0.

### ✨ What's New

* Added **inspect Mithril sync with a dedicated debug server:** Operators can expose a dedicated debug server during Mithril sync to investigate performance and runtime behavior more easily.

### 💪 Improvements

* Improved **tune API backfill to fit each deployment more precisely:** Operators can set the backfill batch size through configuration or flags to better match local storage and workload needs.
* Enhanced **recover stalled Mithril downloads with less wasted progress:** Dingo now detects idle snapshot downloads, validates retry settings early, and resumes partial transfers instead of starting over.
* Refined **reduce memory pressure during dense backfill periods:** API backfill now flushes work based on accumulated rows as well as block count, which keeps heavy historical imports steadier.

### 🔧 Fixes

* Corrected **keep live sync moving when the node reaches the chain tip:** Nodes now avoid a race near the tip that could leave live chain progress waiting for a later update.

### 📋 What You Need to Know

* Clarified **configure backfill batch size when API history import needs different pacing:** Operators can use the new backfill batch size setting in configuration or flags when a smaller or larger block window fits the environment better.
* Highlighted **use Mithril idle retry controls when downloads pause without finishing:** Mithril sync now supports settings that detect stalled downloads, control retry behavior, and continue from partial archive data.
* Emphasized **open a dedicated debug endpoint during Mithril troubleshooting:** Mithril sync can expose debug profiling endpoints on a separate port when deeper runtime inspection is needed.
* Summarized **expect steadier backfill behavior during dense historical imports:** Dingo now flushes accumulated metadata work earlier during heavy batches to reduce memory pressure.
* Noted **expect smoother live follow behavior near the chain tip:** Nodes now continue waiting for the next block more reliably when sync reaches the current tip.
